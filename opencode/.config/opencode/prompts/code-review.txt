Act as a senior software engineer performing a code quality and maintainability review.

Analyze the codebase and identify cleanup opportunities, but do not modify or delete files.

Focus on:

1. Dead code: unused functions, files, components, routes, APIs, variables, imports, and dependencies
2. Duplicate logic that should be consolidated
3. Unused UI components
4. Overly complex implementations that can be simplified
5. Legacy code that appears no longer needed
6. Redundant database queries or API calls
7. Files that appear abandoned or disconnected from the application
8. Opportunities to reduce technical debt

For every finding, include:

* File path and relevant symbol/function/component name
* Why it appears unnecessary or problematic
* Evidence used to determine this, such as references, imports, route usage, tests, grep results, or dependency usage
* Confidence level: high / medium / low
* Impact of removing or simplifying it
* Risks before deletion
* Recommended cleanup plan
* Whether it is safe to remove now or requires manual verification

Important rules:

* Be aggressive in finding cleanup opportunities, but conservative in deletion recommendations.
* Do not claim something is unused unless you checked for references.
* Be careful with dynamic imports, reflection, generated files, framework conventions, route-based files, config files, migrations, scripts, and public APIs.
* Prefer small, safe cleanup steps over large risky rewrites.
* Do not suggest purely stylistic changes unless they reduce complexity or maintenance cost.

Output format:

1. Executive summary
2. High-confidence removals
3. Medium-confidence cleanup candidates
4. Duplicate or redundant logic
5. Simplification opportunities
6. Risky or unclear areas requiring manual review
7. Suggested cleanup order
